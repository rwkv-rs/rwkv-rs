# 异常策略

`rwkv-rs` 将错误分成两类:

## 需要调用方处理的错误

这类错误来自外部世界, 包括:

- 配置缺失
- 输入参数非法
- 网络 / 文件 / 数据库 I/O 失败
- 上游服务不可用

这类路径应通过 `Result` 向上传递.

## 直接暴露内部不变量被破坏的错误

当代码依赖的前提在理论上必须成立时, 使用 `panic` 更合适. 例如:

- tensor shape 与 kernel 契约不一致
- “已经预归一化”的参数再次越界
- 不可达分支实际被走到

此时错误消息必须直接指出:

- 哪个不变量失败
- 实际拿到了什么值
- 开发者应先检查哪个调用点

`panic!` / `expect` / `unwrap` 在实现中都可能有合理用途. 关键不是机械禁用它们, 而是保证语义上站得住脚, 并且在需要时留下足够清晰的上下文.
